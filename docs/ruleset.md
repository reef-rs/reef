# The Reefer Ruleset

The principles that shape every line of code generated by `cargo reef new`. Every rule is here because we've been burned by violating it.

---

### 1. Code cleanliness is next to godliness.

Delete more than you write. The smallest correct codebase wins.

If you're adding a layer of abstraction, ask whether the layer below it could just expose the right API. If you're adding a wrapper, ask whether the wrapped thing was already fine. If you're adding a fallback, ask whether the failure mode is real or hypothetical.

The 40% you delete isn't functionality — it's glue between things that shouldn't have been separate.

---

### 2. Deployment is a hardware distinction, not a software one.

The same binary runs in the cloud, on a customer's edge box, in a thick client, on iOS. **You don't fork your code per target.** Behavior changes via configuration and `--mode` flags, not via separate codebases.

If you find yourself writing "the cloud version" and "the on-prem version" as separate things, you've already lost.

---

### 3. One binary per role, not per environment.

`reef --mode=cloud`, `reef --mode=edge`, `reef --mode=client` should all be the same `reef` binary. Roles change at runtime; artifacts don't.

This makes deployment trivial (one CI artifact), debugging trivial (one symbol set), and rollback trivial (one version number).

---

### 4. Offline is a first-class state, not an error.

Apps must work disconnected. Sync when reconnected. Local writes have to commit locally before the network round-trip, or the UX breaks the moment the WAN coughs.

If your "offline mode" is "show an error toast and disable the form," you don't have an offline mode.

---

### 5. Trust the type system, not the README.

If an invariant matters, encode it in a type. Newtype wrappers, enums for state machines, lifetime annotations for ownership. Compilation should fail before tests do.

Comments lie. Documentation drifts. Types stay current because the compiler enforces them.

---

### 6. Identity at L3, not L7.

Service-to-service authorization belongs at the network layer (Tailnet ACLs), not in per-endpoint guard clauses scattered across HTTP handlers. The handler should never have to ask "is this request allowed?" — the network should already have answered.

Per-endpoint authz is a 2010-era pattern. Stop reinventing it.

---

### 7. The wire format is the contract.

Typed RPC over typed primitives. Dioxus server functions, not OpenAPI specs you write *because* you didn't write Rust.

The schema should be a side effect of the code, not the other way around.

---

### 8. Dependencies are debt.

Every dep is something you don't fully understand, can't fully audit, and didn't write. They're necessary — Reef itself is a curated combination of them — but they're not free.

Audit ruthlessly. Pin versions. Read changelogs before upgrading. If a dep can be replaced with 50 lines of your own code, replace it.

---

### 9. Vendors come and go; standards remain.

- **SQLite** over [proprietary database]. SQLite will outlive every cloud DB vendor.
- **OIDC** over [home-grown auth]. The protocol is older than your company will be.
- **Wireguard** over [VPN-of-the-month]. It's in the kernel.
- **HTTP/2** over [proprietary RPC framework].
- **TOML** over [proprietary config format].

If a piece of infrastructure can be replaced by an open standard, it should be.

---

### 10. Async is the default; sync is the special case.

Tokio is non-negotiable. If you're reaching for `block_on`, ask why. The places where sync makes sense (CPU-bound work in Rayon, single-shot CLI tools) are exceptions, not the rule.

---

### 11. Compile times are a tax; pay them at CI, not at every keystroke.

Workspace structure with small crates. Incremental compilation. Hot reload via Dioxus's dev server. `cargo check` over `cargo build`. Skip release builds in dev.

A 30-second feedback loop kills productivity. A 3-second one doesn't.

---

### 12. `unsafe` and `Box<dyn Trait>` should make you stop and think.

Both have legitimate uses — FFI, plugin systems, dynamic dispatch with stable ABI requirements. Most uses aren't those. If you reach for either, you should be able to articulate *why* concrete generics or borrowed references don't work.

The rule isn't "never use them." The rule is "be deliberate."

---

### 13. Default to a single crate. Drive everything through `dx`.

The default for any new project is **one crate**, with `dx` (not `cargo`) as the only build tool you invoke. This handles every scenario without splitting:

- **Multiple binaries with different content** → `[[bin]]` entries in `Cargo.toml` with `required-features`. `dx serve --bin <name>` selects one and auto-applies its features. Code under `#[cfg(feature = "...")]` doesn't compile when off, so binaries that don't enable a feature cannot contain its code.
- **Different compile targets (WASM vs native)** → feature-gated optional deps + cfg gates. `dx serve --web` orchestrates both compile passes from the same source.
- **Different feature mixes per build** → `dx serve --features "x y"` or `dx serve --no-default-features --features ...`.
- **Code organization** → modules (`pub mod`, `pub(crate)`, `pub(in path)`).

**The Dioxus docs on this:**

> *"If you so choose, you can split your frontend and backend into separate crates. This can be useful for larger projects where you have separate complex entrypoints for web, desktop, and mobile. Workspace setups bring extra complexity..."*

So splitting is opt-in; the docs treat single-crate as the default by example.

**A separate crate (workspace) earns its keep only when:**

1. **Publishing externally** — crates.io for third-party reuse
2. **Compile parallelism** — your codebase is large enough that one-crate compile time hurts iteration
3. **Audit clarity** — "is X in this binary?" needs to be answerable from one Cargo.toml line, not grep-ing cfg gates (useful for security-critical boundaries)
4. **Source-secrecy** — source must live in a separate (often private) repo, which forces a crate split

**Never invoke `cargo build` or `cargo run` directly.** Features are applied per target only when dx orchestrates the build; doing it by hand will get the WASM/native split wrong and silently break compilation. dx is the single tool for building, serving, testing, and bundling.

**The older Rust-monorepo instinct** to split by domain (`edge-vms`, `edge-bridge`, `backend-control`, `storage`, etc.) was load-bearing when projects had two DB engines or two compile-target architectures sharing some crates and not others. With Dioxus-fullstack's "one source, multi-target via features and bin selection" model, the domain split is mostly inertia.

---

## How to apply

The Reefer Ruleset is descriptive, not prescriptive. The scaffolder generates code that follows these rules. As you build on top, you'll feel friction when you violate one — that's intentional.

When in doubt, delete code. When still in doubt, delete more code.
